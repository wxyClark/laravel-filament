# REFLECTION.md - 提示词规律与改进机制

## 用户提示词规律分析

### 1. 决策模式

| 模式 | 示例 | 特点 |
|------|------|------|
| **直接决策** | `方案A`、`执行`、`继续` | 简洁、果断、不犹豫 |
| **结构化决策** | `先执行方案1，然后执行方案2` | 有顺序、有优先级 |
| **问题导向** | `登录又失败了 这是什么问题导致的` | 直接描述问题，期望快速定位 |

### 2. 效率偏好

- 偏好**简短命令**，不喜欢冗长解释
- 希望**一步到位**，减少来回沟通
- 重视**可执行结果**，而非理论分析
- 希望**本地运行**（宿主机），不仅限于 Docker

### 3. 质量意识

- 要求测试通过（`107 passed`）
- 要求代码规范（Pint + PHPStan）
- 要求文档同步更新（README.md）
- 要求**幂等性**（seeder 重复执行不报错）

### 4. 常见痛点

| 问题 | 频率 | 根因 |
|------|------|------|
| Admin 登录失败 | **极高** | DB 被 migrate:fresh 清空 |
| Seeder 内存不足 | 高 | 44k+ 记录 + Telescope 监控 |
| 目录权限问题 | 中 | Docker root vs 宿主机用户 |
| PHP 扩展缺失 | 中 | 本地开发环境不完整 |
| Livewire 数据丢失 | 低 | Collection 序列化问题 |

## 改进机制

### 1. 反思日志

每次重要交互后，记录：
- 用户的原始提示词
- 我的理解和执行
- 结果是否符合预期
- 改进点

### 2. 提示词优化规则

```
✅ 好的提示词：
- "执行" → 明确、无歧义
- "方案A" → 有选项时直接选择
- "先X然后Y" → 有顺序的多步操作

⚠️ 需要补充的信息：
- "测试有问题" → 应补充具体错误信息
- "登录失败" → 应补充是否看到错误页面
```

### 3. 代码生成规范

每次生成代码前，检查：
- [ ] 是否添加 `declare(strict_types=1)`
- [ ] 是否使用 `firstOrCreate` 而非 `create`（幂等性）
- [ ] 是否考虑内存限制（大批量操作）
- [ ] 是否需要更新测试
- [ ] 是否需要更新文档

### 4. 问题排查顺序

```
1. 检查 DB 数据是否存在
2. 检查配置是否正确（auth、route）
3. 检查权限（目录、文件）
4. 检查依赖（PHP 扩展、Composer 包）
5. 检查缓存（Redis、View、Config）
```

### 5. 高频问题防复发

| 问题 | 防复发措施 |
|------|------------|
| migrate:fresh 清空数据 | 创建 `app:reset` 命令 + README 警告 |
| Seeder 内存不足 | 命令自动设置 1G + 禁用 Telescope |
| 目录权限问题 | README 记录修复命令 |
| **擅自删除数据** | **铁律：任何删除操作必须先向用户确认** |
| 地址数据恢复慢 | 创建快照文件 + `--snapshot` 选项（秒级完成） |

### 6. 铁律（不可违反）

- **禁止擅自删除数据库数据**（migrate:fresh、truncate、drop、forceDelete）
- **禁止擅自清空缓存/会话**
- **确认格式**：`即将执行 [操作]，将删除 [具体数据]，是否继续？`
- **例外**：测试环境 SQLite :memory: 的 RefreshDatabase 不需要确认
