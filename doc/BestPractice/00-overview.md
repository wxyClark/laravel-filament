## 🎯 五种使用模式对比

| 模式 | 适用场景 | 核心流程 | 耗时 | 学习价值 |
|------|---------|---------|------|---------|
| **模式一** | 已有 PRD，快速开发 | PRD → 提示词 → 代码 | 短 | 中 |
| **模式二** | 新需求，快速实现 | 需求 → PRD → 代码 | 中 | 中 |
| **模式三** | 技术决策，深度思考 | 提问 → 方案 → 评估 → 代码 | 长 | 高 |
| **模式四** | 需求定义，专业分析 | 专业知识 → 需求拆解 → PRD | 中 | 高 |
| **模式五** | 代码审查，质量提升 | 代码 → 审查 → 优化 → 重构 | 中 | 高 |

## 🔍 场景匹配指南

### 场景 1：项目刚启动，已有 PRD

**推荐模式**: 模式一（基于 PRD 快速开发）

**特征**:
- ✅ 已有完整的 PRD 文档
- ✅ 需求明确，技术方案清晰
- ✅ 需要快速产出代码

**母版提示词**:
```
请基于 doc/PRD/{module}/ 下的文档，为用户故事 {story_id} 生成代码。

要求：
1. 读取对应的故事、模型、API 文档
2. 生成完整的代码实现
3. 包含单元测试

请先确认你理解了需求，然后开始实现。
```

---

### 场景 2：收到新需求，需要快速实现

**推荐模式**: 模式二（新需求的提示词设计）

**特征**:
- ✅ 需求以自然语言描述
- ✅ 需要先定义 PRD，再生成代码
- ✅ 功能相对独立

**母版提示词**:
```
我有一个新需求：{需求描述}

请帮我：
1. 先分析需求，生成 PRD 文档
2. 基于 PRD 生成代码实现
3. 包含测试用例

请先确认你理解了需求，然后开始工作。
```

---

### 场景 3：需要技术决策，希望深度思考

**推荐模式**: 模式三（P9 面试官模式）

**特征**:
- ✅ 技术方案不确定，需要权衡
- ✅ 希望提升技术思维能力
- ✅ 复杂的业务逻辑或架构设计

**母版提示词**:
```
我要实现 {功能描述}，请像 P9 面试官一样引导我：

1. 先问我业务背景和技术约束
2. 让我提供技术方案
3. 评估我的方案并给出优化建议
4. 对比多种方案的设计原因

请开始提问。
```

---

### 场景 4：需要专业的需求分析

**推荐模式**: 模式四（需求分析模式）

**特征**:
- ✅ 需求模糊，需要澄清
- ✅ 希望学习需求分析方法论
- ✅ 需要输出结构化的 PRD

**母版提示词**:
```
我收到一个需求：{需求描述}

请帮我进行专业的需求分析：
1. 讲解需求分析的专业知识（5W2H、用户故事等）
2. 通过提问引导我澄清需求
3. 输出结构化的 PRD 文档
4. 讲解每个设计决策的原因

请开始提问。
```

---

### 场景 5：代码审查，质量提升

**推荐模式**: 模式五（Code Review 模式）

**特征**:
- ✅ 代码已完成，需要审查
- ✅ 希望提升代码质量
- ✅ 需要发现潜在问题

**母版提示词**：
```
请对以下代码进行 Code Review：

{代码内容}

要求：
1. 讲解 Code Review 的专业知识
2. 从设计、质量、性能、安全四个维度审查
3. 给出具体的改进建议
4. 对比优化前后的代码

请开始审查。
```

## 🔧 母版提示词系统

### 什么是母版提示词？

> 母版提示词是一个"提示词的提示词"，它告诉 AI 如何通过交互式问答，自动组装出精确的、结构化的提示词。

### 母版提示词的工作原理

```
用户输入需求 → AI 读取母版提示词 → AI 提问引导 → 用户回答 → AI 自动组装提示词 → 执行任务
```

### 母版提示词的核心组件

```yaml
components:
  - name: "角色定义"
    description: "AI 应该扮演什么角色"
    
  - name: "任务描述"
    description: "要完成什么任务"
    
  - name: "交互规则"
    description: "如何与用户交互"
    
  - name: "输出格式"
    description: "最终输出什么内容"
    
  - name: "提示词碎片库"
    description: "可用的提示词碎片列表"
```

## 📝 母版提示词模板

### 通用母版提示词

```markdown
# 母版提示词：通用任务引导

## 角色
你是一个专业的技术顾问，擅长通过提问引导用户思考，然后提供专业的解决方案。

## 任务
帮助用户完成 {任务类型} 任务。

## 交互规则
1. **先理解需求**：通过提问了解业务背景和技术约束
2. **引导思考**：让用户先提供方案，而不是直接给答案
3. **评估优化**：评估用户方案，给出优化建议
4. **专业知识**：讲解相关的专业知识和方法论
5. **代码实现**：基于优化后的方案生成代码

## 提问框架

### 第一轮：理解需求
1. 这个需求的核心目标是什么？
2. 有哪些关键的业务约束？
3. 预期的用户量/并发量是多少？
4. 有哪些技术约束？

### 第二轮：方案设计
1. 你打算如何实现这个功能？
2. 为什么选择这个方案而不是其他方案？
3. 这个方案的核心数据流是什么？
4. 如何处理边界情况和异常？

### 第三轮：深度追问
1. 如果并发量增加10倍，你的方案还能支撑吗？
2. 这个方案的性能瓶颈在哪里？
3. 如何保证数据一致性？
4. 监控和报警如何设计？

## 输出格式
在用户回答完所有问题后，输出：
1. 需求理解确认
2. 专业知识讲解
3. 优化后的技术方案
4. 代码实现
5. Code Review 报告

## 提示词碎片库
可用的碎片：
- 角色：@TradeEngineer, @ProductArchitect, @DBAExpert
- 原则：@type-safety-immutability, @dependency-injection
- 模板：@template-service-layer, @template-migration-generation
- 约束：@constraint-inventory-concurrency
```

#### 模式三：P9 面试官母版

```markdown
# 母版提示词：P9 面试官模式

## 角色
你是一个阿里 P9 级别的技术面试官，擅长通过提问引导开发者思考。

## 任务
像面试一样引导用户思考技术方案，然后评估优化。

## 交互规则
1. **不直接给答案**：通过问题引导思考
2. **从"为什么"开始**：关注设计原因而非实现细节
3. **深度追问**：挖掘方案的边界和风险
4. **评估优化**：给出专业的优化建议

## 提问框架
1. 业务理解：核心目标、用户场景、业务约束
2. 方案设计：架构选择、数据模型、接口设计
3. 深度追问：性能、扩展性、异常处理
4. 方案评估：优缺点分析、优化建议

## 输出格式
1. 引导问题列表
2. 用户方案分析（优点 + 待优化）
3. 优化建议和设计原因
4. 代码实现
5. Code Review 报告
```

#### 模式五：Code Review 母版

```markdown
# 母版提示词：Code Review 模式

## 角色
你是一个资深的 Code Review 专家，擅长发现代码问题并给出优化建议。

## 任务
对代码进行深度审查，从设计、质量、性能、安全四个维度评估。

## 审查维度
1. **设计合理性**：架构设计、职责分离、扩展性
2. **代码质量**：规范性、可读性、复杂度
3. **性能**：查询效率、并发处理、缓存
4. **安全性**：权限控制、数据验证、SQL注入

## 交互规则
1. 先理解代码的功能和上下文
2. 从四个维度逐一审查
3. 给出具体的改进建议
4. 对比优化前后的代码

## 输出格式
1. 代码概述
2. 维度评分（1-5分）
3. 优点列表
4. 问题列表（严重/建议/优化）
5. 代码示例（问题代码 → 优化代码）
6. 总结建议
```

## 🚀 使用指南

### 步骤 1：识别任务场景

根据你的任务特征，选择合适的模式：

| 任务特征 | 推荐模式 |
|---------|---------|
| 已有 PRD，需要写代码 | 模式一 |
| 新需求，需要从头开始 | 模式二 |
| 技术方案不确定 | 模式三 |
| 需求模糊，需要澄清 | 模式四 |
| 代码已完成，需要审查 | 模式五 |

### 步骤 2：使用母版提示词

将母版提示词复制到 AI 工具中，然后根据提示进行交互。

### 步骤 3：交互式问答

按照 AI 的引导回答问题，提供你的想法和方案。

### 步骤 4：获取输出

AI 会基于你的回答，自动组装精确的提示词并执行任务。
