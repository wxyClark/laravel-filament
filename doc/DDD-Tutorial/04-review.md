# 04 需求评审（评审官 + 架构师）

> 阶段目标：对 03 的需求做可实施性、一致性、风险评审，产出决议。
> 参与：评审官（挑不一致）、架构师（落地成本）。

## 4.1 评审问题清单

| # | 问题 | 决议 |
|---|------|------|
| Q1 | 套餐库存如何管理？ | 套餐不独立占库存，下单按组成 SKU 扣减（见 2.2） |
| Q2 | 虚拟码密码是否落库明文？ | 否，密码用 `Hash::make` 加密，仅展示一次 |
| Q3 | 余额并发安全？ | 支付/充值走数据库事务 + 行锁（`lockForUpdate`） |
| Q4 | 购物车游客态是否需要持久化？ | 本次仅登录态购物车；游客态留扩展点 |
| Q5 | 运费如何算？ | 实体订单固定运费规则（按首重占位），可扩展 |
| Q6 | 售后是否支持部分退款？ | 支持按订单项部分退款（从简：整单或按项） |

## 4.2 范围确认

- 保留：7 模块全做，购物车/售后从简。
- 推迟：游客购物车、三方支付、优惠券系统、拆单、物流轨迹推送。
- 扩展点：PaymentGateway 策略、FulfillmentStrategy（实体/虚拟）、运费策略。

## 4.3 架构师意见

- 聚合根建议：`Product`(SPU)、`Cart`、`Order`、`Payment`(或 PaymentRecord)、`AfterSale`、`Fulfillment`。
- `Sku` 为 `Product` 聚合内实体；`OrderItem` 为 `Order` 聚合内实体。
- 跨聚合协作通过**领域服务 + 领域事件**（如 OrderPaid → Fulfillment 自动建单）。
- 金额统一使用 `decimal(10,2)` 字段 + 整数分运算思路（展示层格式化）。

## 4.4 评审结论

需求通过，进入架构设计（05）。开发顺序：商品 → 购物车 → 订单 → 支付 → 履约 → 售后。
（售后依赖订单与支付，放在最后。）
