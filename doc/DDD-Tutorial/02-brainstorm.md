# 02 头脑风暴（多智能体发散）

> 阶段目标：在需求确定前，尽可能发散地探讨「通用商品交易域」可能涉及的概念、边界与风险。
> 参与智能体：`brainstorm-explore`（发散发散）、`brainstorm-risk`（挑刺）、`brainstorm-arch`（落地性）。

## 2.1 发散清单（brainstorm-explore）

- **商品形态**：单品 / 套餐（组合）/ 虚拟（卡密、授权码、会员）/ 实体（实物）。
- **价格维度**：SKU 级价、会员价、套餐总价、原价/现价。
- **库存**：实体需库存扣减；虚拟为"库存=可售码量"。
- **购物车**：登录态 vs 游客态、跨端合并、失效 SKU 处理。
- **订单**：草稿→待支付→已支付→已完成；拆单、改价、优惠分摊。
- **支付**：余额、充值、退款原路/余额退、支付超时关闭。
- **售后**：退款、退货退款、换货；售后触发库存回滚或虚拟码作废。
- **履约**：实体物流单号、虚拟码自动交付、发货后不可轻易撤销。

## 2.2 风险与质疑（brainstorm-risk）

- 套餐与 SKU 的库存关系容易错：套餐不应独立占库存，应按组成 SKU 扣减。
- 虚拟码"唯一码+密码"需防泄露：密码应加密存储或仅一次性展示。
- 余额支付与充值要防并发超扣：需事务 + 行锁。
- 订单金额计算必须幂等，避免重复支付导致多次扣款。

## 2.3 落地性（brainstorm-arch）

- 现有项目已具备 DDD 分层与架构测试，可直接在 `app/Domains` 下新建 `Catalog/Cart/Order/Payment/AfterSale/Fulfillment`。
- 金额统一 `decimal(10,2)`，复用既有 `Cache::remember` 缓存风格。
- 虚拟码交付可独立成 `Fulfillment` 聚合，避免污染 Order。

## 2.4 收敛结论

- 7 模块按 SPEC-MASTER 执行，购物车/售后从简。
- 虚拟交付形态固定为「唯一码 + 密码」，加密存储密码。
- 支付仅余额 + 充值，其余留策略扩展点。
