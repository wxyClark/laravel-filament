## 📚 专业知识讲解

### 1. 数据库设计的核心概念

**什么是数据库设计？**
> 数据库设计是将业务需求转化为数据库结构的过程，包括表结构设计、索引设计、关系设计等。

**数据库设计的目标：**
```
数据完整性   → 数据准确、一致
数据冗余最小 → 避免重复存储
查询效率高   → 响应速度快
可维护性好   → 易于修改和扩展
```

**数据库设计的范式：**
| 范式 | 说明 | 目标 |
|------|------|------|
| **1NF** | 字段不可再分 | 原子性 |
| **2NF** | 消除部分依赖 | 完全依赖 |
| **3NF** | 消除传递依赖 | 直接依赖 |
| **BCNF** | 消除主属性传递依赖 | 更严格 |

### 2. 表结构设计

**字段类型选择：**
| 数据 | 类型 | 示例 | 说明 |
|------|------|------|------|
| 整数 | INT/BIGINT | user_id | 根据范围选择 |
| 金额 | DECIMAL | price | 精确计算 |
| 字符串 | VARCHAR | name | 变长 |
| 文本 | TEXT | description | 长文本 |
| 时间 | TIMESTAMP | created_at | 日期时间 |
| 布尔 | TINYINT | is_active | 0/1 |

**主键设计：**
```
自增主键: id BIGINT AUTO_INCREMENT
- 优点: 简单、有序、索引友好
- 缺点: 分布式环境不友好

UUID: id CHAR(36)
- 优点: 全局唯一、分布式友好
- 缺点: 无序、索引不友好
```

**外键设计：**
```php
// Laravel 迁移示例
$table->foreignId('user_id')
      ->constrained()
      ->onDelete('cascade');
```

### 3. 索引设计

**索引类型：**
| 类型 | 说明 | 适用场景 |
|------|------|---------|
| **主键索引** | 唯一标识 | 每个表必须 |
| **唯一索引** | 值唯一 | 邮箱、手机号 |
| **普通索引** | 加速查询 | 常用查询字段 |
| **复合索引** | 多字段组合 | 联合查询 |

**索引设计原则：**
```
1. 在 WHERE 子句中的字段上创建索引
2. 在 JOIN 子句中的外键上创建索引
3. 在 ORDER BY 子句中的字段上创建索引
4. 避免在低基数字段上创建索引
5. 避免在频繁更新的字段上创建索引
```

**索引失效场景：**
```
1. 使用函数或表达式: WHERE YEAR(created_at) = 2024
2. 使用 LIKE 左模糊: WHERE name LIKE '%abc'
3. 使用 OR 条件（部分场景）
4. 使用 NOT IN / NOT EXISTS
5. 数据类型不匹配
```

### 4. 关系设计

**关系类型：**
| 类型 | 说明 | 示例 |
|------|------|------|
| **一对一** | 1:1 | 用户 - 用户资料 |
| **一对多** | 1:N | 用户 - 订单 |
| **多对多** | M:N | 文章 - 标签 |

**外键策略：**
```
CASCADE: 级联删除（删除父记录时删除子记录）
RESTRICT: 限制删除（有子记录时禁止删除父记录）
SET NULL: 置空（删除父记录时将子记录外键置空）
NO ACTION: 无操作（默认）
```

## 💡 知识点总结

### 数据库设计的核心原则

1. **规范化**：遵循范式减少数据冗余
2. **反规范化**：适当冗余提高查询效率
3. **索引优化**：为常用查询创建索引
4. **数据完整性**：使用约束保证数据准确
5. **可扩展性**：预留扩展字段或分表方案

### 数据库设计的检查清单

- [ ] 表结构是否满足范式？
- [ ] 字段类型是否合适？
- [ ] 主键设计是否合理？
- [ ] 外键关系是否正确？
- [ ] 索引设计是否优化？
- [ ] 注释是否完整？
- [ ] 数据量预估是否合理？
