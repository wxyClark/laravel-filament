# 任务模板：Code Review 审查

## 用途说明
对代码进行深度审查，发现设计问题、性能问题、安全隐患，并给出改进建议。

## 适用场景
- 代码提交前的自我审查
- 团队代码评审
- 架构设计评审

---

## 标准审查流程

### 阶段 1：代码理解

```markdown
## 📖 代码理解

### 1.1 代码概述
- 模块名称: {module_name}
- 主要功能: {功能描述}
- 代码行数: {行数}
- 复杂度: {高/中/低}

### 1.2 核心类/方法
| 类/方法 | 职责 | 行数 |
|---------|------|------|
| {类名} | {职责} | {行数} |
| {方法名} | {职责} | {行数} |

### 1.3 依赖关系
- 依赖的类: {列表}
- 被依赖的类: {列表}
- 外部依赖: {包/服务}
```

### 阶段 2：设计评审

```markdown
## 🏗️ 设计评审

### 2.1 架构设计
- 设计模式: {使用的模式}
- 分层架构: {是否遵循}
- 职责分离: {评估}

### 2.2 SOLID 原则检查
| 原则 | 是否遵循 | 说明 |
|------|---------|------|
| 单一职责 | ✅/❌ | {说明} |
| 开闭原则 | ✅/❌ | {说明} |
| 里氏替换 | ✅/❌ | {说明} |
| 接口隔离 | ✅/❌ | {说明} |
| 依赖倒置 | ✅/❌ | {说明} |

### 2.3 设计问题
{识别到的设计问题和改进建议}
```

### 阶段 3：代码质量检查

```markdown
## 🔍 代码质量检查

### 3.1 命名规范
| 检查项 | 结果 | 说明 |
|--------|------|------|
| 类名是否清晰 | ✅/❌ | {说明} |
| 方法名是否清晰 | ✅/❌ | {说明} |
| 变量名是否清晰 | ✅/❌ | {说明} |

### 3.2 代码复杂度
| 指标 | 值 | 评估 |
|------|-----|------|
| 方法平均行数 | {值} | {好/一般/差} |
| 最长方法行数 | {值} | {好/一般/差} |
| 圈复杂度 | {值} | {好/一般/差} |

### 3.3 重复代码
- 重复代码块数: {数量}
- 重复率: {百分比}

### 3.4 代码问题
{识别到的代码质量问题}
```

### 阶段 4：性能审查

```markdown
## ⚡ 性能审查

### 4.1 数据库查询
| 检查项 | 结果 | 说明 |
|--------|------|------|
| N+1 查询 | ✅/❌ | {说明} |
| 缺少索引 | ✅/❌ | {说明} |

### 4.2 缓存使用
- 是否有缓存: {是/否}
- 缓存策略: {策略}
- 缓存命中率: {评估}

### 4.3 并发处理
- 并发场景: {场景}
- 锁策略: {策略}
- 死锁风险: {评估}

### 4.4 性能问题
{识别到的性能问题}
```

### 阶段 5：安全审查

```markdown

### 5.1 常见漏洞

### 5.2 数据验证
- 输入验证: {评估}
- 输出编码: {评估}

### 5.3 安全问题
{识别到的安全问题}
```

### 代码示例

{有问题的代码}
```

// 建议实现
{优化后的代码}

{总结性评价和建议}
```
