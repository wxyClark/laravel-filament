# 任务模板：需求分析与定义

> **v2.0: 专业知识讲解 + 面试引导**

## 用途说明
规范需求分析过程，帮助开发者理解需求分析的专业知识和方法论。

## 适用场景
- 接收新需求时的需求分析
- 需求评审前的准备工作
- 提升需求分析能力

## 📚 专业知识讲解

### 1. 需求分析的核心概念

**什么是需求分析？**
> 需求分析是将用户的业务需求转化为技术实现方案的过程。它不仅仅是"做什么"，更重要的是"为什么这样做"。

**需求分析的层次：**
```
业务需求 → 用户需求 → 系统需求 → 技术需求
    ↓           ↓           ↓           ↓
  为什么      做什么       怎么做       用什么做
```

**需求分析的关键要素：**
| 要素 | 说明 | 示例 |
|------|------|------|
| **目标** | 为什么要做这个需求 | 提升用户转化率 |
| **用户** | 谁会使用这个功能 | C端用户、管理员 |
| **场景** | 在什么情况下使用 | 下单场景、支付场景 |
| **规则** | 业务约束和逻辑 | 库存扣减规则 |
| **边界** | 不做什么 | 不处理退款 |

### 2. 需求分析的方法论

**5W2H 分析法：**
```
What    - 是什么？做什么？
Why     - 为什么做？
Who     - 谁来做？谁用？
When    - 什么时候做？截止时间？
Where   - 在哪里做？范围？
How     - 怎么做？
How much - 做多少？成本？
```

**用户故事三要素：**
```
作为 <角色>
我想要 <功能>
以便 <价值>
```

**验收标准（Given-When-Then）：**
```
Given <前置条件>
When  <触发操作>
Then  <预期结果>
```

### 3. 需求分析的常见陷阱

| 陷阱 | 说明 | 解决方案 |
|------|------|---------|
| **需求模糊** | "做个好用的系统" | 量化需求，明确指标 |
| **需求镀金** | 过度设计 | 聚焦核心需求 |
| **需求蔓延** | 范围不断扩大 | 明确边界 |
| **假设需求** | 没有验证就开发 | 用户访谈验证 |

## 🎯 面试引导流程

### 第一轮：理解业务背景

```
1. 这个需求的业务目标是什么？
   - 核心KPI是什么？
   - 预期提升多少？

2. 目标用户是谁？
   - 用户画像是什么？
   - 用户量级是多少？

3. 竞品是怎么做的？
   - 行业标准是什么？
   - 我们的差异化在哪里？
```

### 第二轮：明确需求边界

```
4. 核心功能是什么？
   - MVP 包含哪些功能？
   - 哪些可以后续迭代？

5. 不做什么？
   - 明确排除的功能
   - 技术限制

6. 预期时间线？
   - 开发周期
   - 里程碑节点
```

### 第三轮：技术可行性

```
7. 技术约束是什么？
   - 现有架构限制
   - 团队技能限制

8. 性能要求是什么？
   - 并发量
   - 响应时间

9. 安全要求是什么？
   - 数据安全
   - 权限控制
```

## 💡 知识点总结

### 需求分析的核心原则

1. **以用户为中心**：始终从用户角度思考
2. **量化需求**：用数据说话，避免模糊描述
3. **明确边界**：清楚做什么，更清楚不做什么
4. **验证需求**：通过用户访谈验证假设
5. **迭代优化**：需求是演进的，不是一次性的

### 需求分析的检查清单

- [ ] 业务目标是否明确？
- [ ] 目标用户是否清晰？
- [ ] 功能范围是否定义？
- [ ] 验收标准是否可测试？
- [ ] 技术约束是否识别？
- [ ] 时间线是否合理？
- [ ] 风险是否识别？
