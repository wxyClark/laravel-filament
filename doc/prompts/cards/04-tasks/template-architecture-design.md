## 📚 专业知识讲解

### 1. 架构设计的核心概念

**什么是架构设计？**
> 架构设计是将系统需求转化为技术实现方案的过程，定义了系统的组件结构、交互方式和演化方向。

**架构设计的目标：**
```
可维护性 → 代码易于修改和扩展
可扩展性 → 系统能够承载业务增长
高性能   → 系统响应速度快
高可用   → 系统稳定可靠
安全性   → 系统抵御攻击
```

**架构设计的原则：**
| 原则 | 说明 | 示例 |
|------|------|------|
| **单一职责** | 一个模块只做一件事 | 用户模块只管用户 |
| **开闭原则** | 对扩展开放，对修改关闭 | 使用接口扩展 |
| **依赖倒置** | 依赖抽象而非具体 | 依赖接口 |
| **接口隔离** | 接口小而专 | 拆分大接口 |

### 2. 架构模式

**分层架构：**
```
┌─────────────────────────┐
│   Presentation Layer    │  ← 用户界面
├─────────────────────────┤
│   Application Layer     │  ← 业务逻辑
├─────────────────────────┤
│   Domain Layer          │  ← 领域模型
├─────────────────────────┤
│   Infrastructure Layer  │  ← 数据访问
└─────────────────────────┘
```

**六边形架构（端口适配器）：**
```
        ┌─────────────────────┐
        │     Application     │
        │       Service       │
        └──────────┬──────────┘
                   │
    ┌──────────────┼──────────────┐
    │              │              │
┌───▼───┐    ┌────▼────┐    ┌───▼───┐
│ HTTP  │    │ Console │    │ Queue │
│ Adapter│    │ Adapter │    │Adapter│
└───────┘    └─────────┘    └───────┘
```

**DDD 领域驱动设计：**
```
┌─────────────────────────────────────┐
│           Bounded Context           │
├─────────────────────────────────────┤
│  Entity  │  Value Object  │  Aggregate │
├─────────────────────────────────────┤
│  Domain Event  │  Domain Service    │
├─────────────────────────────────────┤
│  Repository Interface  │  Factory   │
└─────────────────────────────────────┘
```

## 🎯 面试引导流程

### 第一轮：理解业务需求

```
1. 这个系统的业务目标是什么？
   - 核心KPI是什么？
   - 预期用户量级？

2. 系统的非功能需求是什么？
   - 性能要求？
   - 可用性要求？
   - 安全要求？

3. 技术约束是什么？
   - 现有技术栈？
   - 团队技能？
   - 预算限制？
```

### 第二轮：设计架构方案

```
4. 选择什么架构模式？
   - 单体？微服务？Serverless？
   - 为什么选择这个模式？

5. 如何划分模块边界？
   - 模块职责是什么？
   - 模块间如何通信？

6. 数据如何存储？
   - 关系型数据库？NoSQL？
   - 缓存策略？
```

### 第三轮：评估架构方案

```
7. 架构的可扩展性如何？
   - 如何应对用户增长？
   - 如何添加新功能？

8. 架构的可维护性如何？
   - 代码复杂度？
   - 文档完整性？

9. 架构的风险是什么？
   - 单点故障？
   - 性能瓶颈？
```

## 💡 知识点总结

### 架构设计的核心原则

1. **简单性**：选择最简单的方案，避免过度设计
2. **演化性**：架构应该能够演进，而不是一次性设计
3. **权衡性**：没有完美的架构，只有适合的架构
4. **文档化**：架构决策必须记录，便于后续维护
5. **验证性**：架构设计需要通过原型验证

### 架构设计的检查清单

- [ ] 业务需求是否理解？
- [ ] 非功能需求是否明确？
- [ ] 架构模式是否选择？

- [ ] 数据存储是否设计？
- [ ] 接口是否定义？
- [ ] 安全措施是否考虑？
- [ ] 性能优化是否规划？
- [ ] 风险是否识别？
- [ ] 决策是否记录？
