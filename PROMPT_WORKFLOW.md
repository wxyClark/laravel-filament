# PROMPT_WORKFLOW.md - 提示词增强工作流

## 标准流程

每次收到用户提示词后，执行以下步骤：

```
用户提示词 → 1.理解 → 2.增强 → 3.确认 → 4.执行 → 5.验证
```

### 步骤 1：理解（多轮对话挖掘）

**目标**: 通过多轮对话深度理解需求，挖掘隐含需求

**对话框架**:

```
Phase 1: 初始理解 (1-2轮)
├── 核心功能是什么？
├── 使用角色有哪些？
└── 有参考来源吗？

Phase 2: 深度挖掘 (3-5轮)
├── 数据生命周期？
├── 权限边界？
├── 异常场景？
├── 性能预期？
└── 集成需求？

Phase 3: 逻辑验证 (2-3轮)
├── 状态机完整性？
├── 权限覆盖？
├── 数据一致性？
├── 并发安全？
└── 边界完整？

Phase 4: 全局一致性 (1-2轮)
├── 术语一致？
├── 数据模型一致？
├── 状态一致？
├── 权限一致？
└── 风格一致？

Phase 5: 文档沉淀 (1轮)
├── 输出需求文档
├── 记录对话摘要
└── 列出待确认事项
```

**追问技巧**:
- 开放式追问: "搜索功能需要支持哪些维度？"
- 场景化追问: "假设两个管理员同时编辑同一个订单..."
- 边界追问: "订单金额可以是负数吗？"
- 依赖追问: "这个功能需要依赖哪些现有模块？"

**逻辑闭环检查**:
- ✅ 起点: 什么触发这个流程？
- ✅ 过程: 流程经过哪些步骤？
- ✅ 终点: 流程如何结束？
- ✅ 异常: 每个步骤失败怎么处理？
- ✅ 回滚: 如何撤销已完成的操作？

### 步骤 2：增强

根据意图，自动读取：

| 意图类型 | 读取内容 |
|----------|----------|
| 功能开发 | PRD 文档、相关 Model、Service、现有测试 |
| 代码修改 | 相关文件、AGENTS.md 代码规范、现有代码风格 |
| 问题修复 | 错误日志、相关配置、测试用例 |
| 测试编写 | 现有测试模式、Pest.php 配置、Factory |

增强为结构化提示词：

```
【目标】一句话描述要做什么
【涉及文件】列出需要修改/创建的文件
【约束条件】技术约束、规范约束
【验收标准】如何验证完成
【风险点】可能出错的地方
```

### 步骤 3：确认

将增强后的提示词呈现给用户确认：

```
我理解你的需求是：

【目标】...
【涉及文件】...
【约束条件】...
【验收标准】...
【风险点】...

确认执行？
```

如有语义模糊，提出具体问题：

```
你的需求有以下模糊点，请选择或补充：
1. "优化性能" → 具体指查询优化、缓存、还是懒加载？
2. "修复问题" → 请提供错误信息或截图
```

### 步骤 4：执行

用户确认后，按增强后的提示词执行：

1. 先读取涉及的文件
2. 按约束条件编写代码
3. 运行 Pint + PHPStan + Pest 验证
4. 添加/更新架构测试（DDD 边界 + 命名约定）

### 步骤 5：验证

执行后自检：

- [ ] 代码是否符合 Pint 规范
- [ ] PHPStan 是否通过
- [ ] 测试是否全部通过
- [ ] 架构测试是否通过（DDD 分层 + 命名约定）
- [ ] 是否需要更新文档

## 提示词增强示例

### 示例 1：功能开发

用户输入：`给地址列表加个搜索功能`

增强后：

```
【目标】为地址列表页（ViewAddressList）添加关键字搜索功能
【涉及文件】
  - app/Filament/Admin/Pages/ViewAddressList.php
  - resources/views/filament/pages/address-list.blade.php
【约束条件】
  - 搜索范围：名称、拼音、行政区划代码
  - 支持模糊匹配
  - 与现有级联筛选联动
  - 搜索时重置分页到第1页
【验收标准】
  - 输入关键字后，表格实时过滤显示匹配结果
  - 空关键字时显示全部
  - Pint + PHPStan + Pest 通过
【风险点】
  - 44k+ 记录的搜索性能
  - 与级联筛选的优先级逻辑
```

### 示例 2：问题修复

用户输入：`登录又失败了`

增强后：

```
【目标】排查并修复管理员登录失败问题
【排查步骤】
  1. 检查 admins 表是否有数据：SELECT COUNT(*) FROM admins
  2. 检查密码哈希是否正确：php artisan tinker
  3. 检查 auth 配置：config('auth.guards.admin')
  4. 检查 Session 驱动是否正常
【涉及文件】
  - config/auth.php
  - app/Models/Admin.php
  - database/seeders/DatabaseSeeder.php
【验收标准】
  - 访问 http://localhost:8080/admin/login 可登录
  - Session 保持正常
```

## 语义模糊处理规则

| 模糊词 | 处理方式 |
|--------|----------|
| "优化" | 追问：性能优化 / 代码优化 / 用户体验优化？ |
| "修复" | 追问：具体错误信息 / 期望行为 / 实际行为？ |
| "添加" | 确认：功能范围 / 技术方案 / 影响范围？ |
| "更新" | 确认：更新内容 / 是否需要数据库变更？ |
| "删除" | 警告：数据安全影响 / 是否需要软删除？ |

---

## 自动触发条件

以下情况必须执行增强流程：

1. 功能开发类需求
2. 涉及数据库变更的需求
3. 涉及认证/授权的需求
4. 涉及性能优化的需求
5. 涉及多个文件修改的需求

以下情况可简化流程：

1. 明确的单文件修改（如 "把这个变量名改一下"）
2. 简单的配置查看（如 "看看当前的路由列表"）
3. 已有明确方案的执行（如 "执行方案A"）
