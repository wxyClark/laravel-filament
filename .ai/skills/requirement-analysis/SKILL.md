# Requirement Analysis — 需求分析与逻辑验证

> **目标**: 通过多轮对话深度挖掘需求，确保逻辑闭环、全局一致、风格统一
> **原则**: 追问 > 假设，验证 > 推测，文档 > 记忆
> **输出**: 需求确认文档 + 逻辑验证报告

---

## 核心原则

```
🔍 多轮追问 → 挖掘隐含需求
🔗 逻辑闭环 → 每个流程有始有终
🌐 全局一致 → 跨模块无冲突
📏 风格统一 → 术语/格式/模式一致
📝 文档沉淀 → 对话成果可追溯
```

---

## 对话流程框架

### Phase 1: 初始理解 (1-2轮)

**目标**: 理解用户的核心诉求

```
用户: 我想做一个订单管理功能

AI 追问:
1. 订单管理包含哪些核心操作？（增删改查、状态流转、批量操作？）
2. 使用角色有哪些？（管理员、普通用户、客服？）
3. 是否有现成的参考？（竞品、截图、原型？）
```

**输出格式**:
```
【初步理解】
- 核心功能: {功能列表}
- 用户角色: {角色列表}
- 参考来源: {参考信息}
```

### Phase 2: 深度挖掘 (3-5轮)

**目标**: 发现隐含需求和边界条件

**追问维度**:

| 维度 | 追问示例 |
|------|----------|
| 数据生命周期 | "订单创建后有哪些状态？每个状态能做什么操作？" |
| 权限边界 | "普通用户能看到其他人的订单吗？" |
| 异常场景 | "如果库存不足怎么办？支付超时怎么处理？" |
| 性能预期 | "预计有多少订单？需要支持实时更新吗？" |
| 集成需求 | "需要对接支付系统吗？需要发送通知吗？" |
| 数据迁移 | "现有数据怎么处理？需要导入历史数据吗？" |

**逻辑闭环检查**:

```
每个业务流程必须回答:
✅ 起点: 什么触发这个流程？
✅ 过程: 流程经过哪些步骤？
✅ 终点: 流程如何结束？
✅ 异常: 每个步骤失败怎么处理？
✅ 回滚: 如何撤销已完成的操作？
```

**示例对话**:

```
AI: 订单状态流转是怎样的？
用户: 待支付 → 已支付 → 已发货 → 已完成

AI: 追问:
1. 有没有"已取消"状态？什么情况下可以取消？
2. "已支付"到"已发货"之间有没有"备货中"？
3. "已完成"后可以退货吗？退货流程是什么？
4. 每个状态转换由谁触发？（用户、管理员、系统？）
5. 状态变更需要通知谁？（短信、邮件、站内信？）
```

### Phase 3: 逻辑验证 (2-3轮)

**目标**: 验证逻辑完整性和一致性

**验证清单**:

| 检查项 | 验证方法 |
|--------|----------|
| 状态机完整性 | 每个状态都有出路和入口 |
| 权限覆盖 | 每个操作都有权限控制 |
| 数据一致性 | 关联数据更新保持一致 |
| 并发安全 | 多人同时操作不冲突 |
| 边界完整 | 空数据、满数据、极端值都能处理 |

**逻辑冲突检测**:

```
检查项:
1. 跨模块数据依赖是否闭环？
2. 同一实体在不同模块的状态是否一致？
3. 并发操作是否有竞态条件？
4. 异步操作是否有回调确认？
5. 缓存数据是否与数据库同步？
```

**示例验证**:

```
AI: 我来验证一下订单创建的逻辑闭环:

【订单创建流程】
1. 用户选择商品 → 检查库存 ✓
2. 提交订单 → 扣减库存 ✓
3. 支付成功 → 更新订单状态 ✓
4. 支付失败 → 释放库存？ ← 需要确认

【发现的问题】
- 支付失败后库存是否自动恢复？
- 库存恢复的时间窗口是多久？
- 如果用户重新支付，库存是否需要重新检查？

请确认这些细节。
```

### Phase 4: 全局一致性 (1-2轮)

**目标**: 确保与现有系统无冲突

**一致性检查**:

| 检查维度 | 检查内容 |
|----------|----------|
| 术语一致 | 同一概念在不同模块的命名是否相同 |
| 状态一致 | 同一实体在不同场景的状态定义是否一致 |
| 权限一致 | 同一角色在不同模块的权限是否冲突 |
| 数据一致 | 关联模块的数据更新是否同步 |
| 风格一致 | 代码/文档/接口的风格是否统一 |

**术语对齐**:

```
现有系统术语:
- 用户: User (用户表)
- 客户: Customer (客户表)
- 会员: Member (会员表)

新模块术语:
- 买家: Buyer

确认: Buyer 是指 User 还是 Customer？还是新的实体？
```

**风格对齐**:

```
现有系统风格:
- 状态字段: 使用 string 类型 + default
- 时间字段: 使用 timestamp 类型
- 金额字段: 使用 decimal(10,2)
- API 响应: { code, message, data, meta }

新模块确认:
- 状态字段: string + default ✓
- 时间字段: timestamp ✓
- 金额字段: decimal(10,2) ✓
- API 响应: { code, message, data, meta } ✓
```

### Phase 5: 文档沉淀 (1轮)

**目标**: 将对话成果转化为可追溯的文档

**输出文档**:

```
doc/requirements/REQ-{模块}-{序号}.md

内容:
1. 需求概述
2. 用户故事
3. 功能清单
4. 逻辑流程图
5. 状态机定义
6. API 契约
7. 异常处理
8. 验收标准
9. 对话记录摘要
10. 待确认事项
```

## 追问技巧

### 开放式追问

```
❌ 封闭式: "需要搜索功能吗？"
✅ 开放式: "搜索功能需要支持哪些维度？（名称、编码、标签？）"

❌ 封闭式: "需要分页吗？"
✅ 开放式: "列表数据量预期是多少？需要什么样的分页体验？"
```

### 场景化追问

```
"假设一个场景：管理员正在处理100个待发货订单，系统突然断电，
恢复后这些订单的状态是什么？需要怎么处理？"

"如果两个管理员同时编辑同一个订单，会发生什么？"
```

### 边界追问

```
"订单金额可以是负数吗？什么场景下会出现负数？"

"订单备注最多能输入多少字？超长怎么处理？"

"如果用户创建订单后不支付，多久自动取消？"
```

### 依赖追问

```
"这个功能需要依赖哪些现有模块？"

"修改这个功能会影响哪些其他功能？"

"需要对接哪些外部系统？（支付、短信、物流？）"
```

```markdown
## {实体} 状态机

### 状态定义
| 状态 | 描述 | 入口 | 出口 |
|------|------|------|------|
| {状态1} | {描述} | {入口条件} | {出口条件} |
| {状态2} | {描述} | {入口条件} | {出口条件} |

### 状态流转图
```
[{状态1}] --{触发条件}--> [{状态2}]
[{状态2}] --{触发条件}--> [{状态3}]
[{状态2}] --{触发条件}--> [{状态4}]
```

### 闭环验证
- [ ] 每个状态都有入口
- [ ] 每个状态都有出口（终态除外）
- [ ] 没有死循环
- [ ] 没有孤立状态
- [ ] 异常状态有处理
```

## 风险识别

### 常见需求风险

| 风险类型 | 识别信号 | 应对措施 |
|----------|----------|----------|
| 需求模糊 | "大概"、"可能"、"应该" | 追问具体场景 |
| 需求变更 | 频繁修改、前后矛盾 | 确认优先级、冻结需求 |
| 隐含需求 | "当然"、"默认"、"不用说" | 追问细节、列出假设 |
| 技术约束 | "系统不支持"、"做不了" | 了解约束、寻找替代 |
| 资源不足 | 时间紧、人手少 | 降低范围、分期交付 |
