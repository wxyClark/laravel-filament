# Prompt Engineer — 提示词优化器

> **目标**: 将用户模糊/简短的输入，优化为结构化、高质量的提示词，让 AI 生成准确代码
> **原则**: 补全上下文 > 猜测意图，明确约束 > 开放发挥，可验证 > 靠谱
> **输出**: 结构化提示词 + 缺失信息清单 + 验证要点

---

## 核心原则

```
🎯 补全上下文 → AI 不需要猜测
📐 明确约束 → 减少歧义和返工
🔗 关联项目 → 符合现有架构
✅ 可验证 → 生成后能立即验证
📝 结构化 → 分步骤、分层次描述
```

## 优化流程

### Phase 1: 分析用户输入

**识别输入类型**:

| 输入类型 | 示例 | 优化方向 |
|----------|------|----------|
| 模糊需求 | "加个搜索功能" | 追问维度 + 添加约束 |
| 片段描述 | "用户表加个字段" | 补全上下文 + 关联影响 |
| 技术问题 | "这个报错怎么修" | 添加错误信息 + 复现步骤 |
| 功能描述 | "实现导出 Excel" | 添加性能要求 + 使用场景 |

**提取关键信息**:

```
从用户输入中提取:
1. 核心动作: 做什么？（增删改查、导入导出、计算、转换...）
2. 涉及实体: 对什么操作？（模型、表、字段、API...）
3. 使用场景: 谁用？在哪用？（页面、接口、命令行...）
4. 隐含约束: 没说但应该有的？（性能、安全、兼容...）
```

### Phase 2: 识别缺失信息

**缺失信息检测清单**:

```
□ 实体定义: 模型/表结构清楚吗？
□ 字段类型: 用什么数据类型？
□ 关联关系: 与其他实体的关系？
□ 权限控制: 谁能操作？
□ 异常处理: 失败怎么处理？
□ 性能要求: 数据量多大？响应时间要求？
□ UI/UX: 界面怎么展示？
□ 测试要求: 需要测试吗？
□ 文档要求: 需要文档吗？
```

**项目特定上下文** (本项目):

```
技术栈:
- Laravel 12 + Filament 3.x
- PHP 8.5+
- MySQL 8.0
- Redis (缓存/队列)
- Docker 环境

架构:
- DDD 分层: Domain / Infrastructure / Http
- 资源位置: app/Filament/Admin/Resources
- 页面位置: app/Filament/Admin/Pages

规范:
- Pint (PSR-12 + strict_types)
- PHPStan level 5
- Pest 测试
- TDD 工作流

数据规范:
- 金额: decimal(10,2)
- 软删除: 核心表开启 SoftDeletes
- 时间: timestamp 类型
```

### Phase 3: 生成结构化提示词

**输出模板**:

```markdown
## 需求概述
{一句话描述核心需求}

## 具体要求

### 1. {模块/功能点 1}
- 目标: {要达到什么效果}
- 输入: {输入数据}
- 输出: {输出结果}
- 约束: {限制条件}

### 2. {模块/功能点 2}
...

## 技术约束
- 文件: {需要创建/修改的文件}
- 依赖: {需要使用的包/服务}
- 性能: {性能要求}

## 验证方式
- [ ] {验证点 1}
- [ ] {验证点 2}
```

## 优化技巧

### 模糊需求 → 具体需求

```
用户: "加个搜索"
↓ 优化
用户: "地址列表页面添加搜索功能，支持按名称、编码、拼音模糊搜索，
      筛选条件与现有的省份/城市/区县/街道级联筛选联动，
      搜索结果实时更新，显示匹配数量"
```

### 简短描述 → 完整描述

```
用户: "导出 Excel"
↓ 优化
用户: "地址管理页面添加数据导出功能，要求：
      1. 支持 CSV 和 Excel 两种格式
      2. 导出当前筛选条件下的全量数据（不仅当前页）
      3. 数据量约 12 万条，需要队列异步处理
      4. 导出列：ID、名称、编码、层级、上级地址、拼音、路径、排序
      5. 导出完成后通知用户下载"
```

### 技术问题 → 可操作需求

```
用户: "这个报错怎么回事"
↓ 优化
用户: "访问 /admin/address/1 时报错 Missing required parameter，
      错误位置: UrlGenerationException:35，
      路由定义: Route: filament.admin.pages.address [URI: admin/address/{id}]
      请分析原因并修复"
```

## 项目上下文注入

### 自动注入的信息

当用户输入不完整时，自动补充:

```
项目上下文:
- 项目: Laravel + Filament 后台管理系统
- PHP: 8.5+ (strict_types)
- 架构: DDD (Domain/Infrastructure/Http)
- 规范: Pint + PHPStan level 5 + Pest

代码规范:
- 所有 PHP 文件必须 declare(strict_types=1)
- 类成员顺序: trait → constant → property → constructor → method
- 使用 short array syntax []
- import 语句按字母排序
- 不写注释，除非 WHY 非显而易见

架构约束:
- Domain 层不依赖框架
- Infrastructure 层实现 Domain 接口
- Http 层只调用 Service
- Filament 资源必须有 form() 和 table() 方法
```

## 使用示例

### 示例 1: 模糊需求

```
用户输入: "加个权限管理"

优化后输出:
## 需求概述
为系统添加 RBAC 权限管理功能

## 具体要求

### 1. 角色管理
- 目标: 管理员可以创建、编辑、删除角色
- 字段: name, guard_name, permissions (多对多)
- 页面: Filament Resource (CRUD)

### 2. 权限管理
- 目标: 定义系统权限点
- 字段: name, guard_name
- 分组: 按模块分组显示

### 3. 用户-角色分配
- 目标: 为用户分配角色
- 页面: 用户详情页添加角色选择器

## 技术约束
- 使用 spatie/laravel-permission (已安装)
- 遵循 DDD 架构
- 需要数据库迁移

## 验证方式
- [ ] 角色 CRUD 正常
- [ ] 权限分配正确
- [ ] 权限检查生效
```

### 示例 2: 技术问题

```
用户输入: "页面报错 500"

优化后输出:
## 问题描述
页面访问返回 500 错误

## 需要信息
- [ ] 完整错误信息 (storage/logs/laravel.log)
- [ ] 访问的 URL
- [ ] 操作步骤（做了什么导致报错）
- [ ] 是否修改过代码

## 初步排查方向
1. 检查 Laravel 日志
2. 检查权限配置
3. 检查路由定义
4. 检查数据库连接
```

